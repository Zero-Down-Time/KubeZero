# ci-tools-lib

Shared CI/CD toolchain for ZeroDownTime: rootless Podman container builds, multi-arch (amd64/arm64), Grype + betterleaks scanning, AWS ECR Public publishing. Consumed as a `git submodule` at `.ci/` inside downstream projects (Renovate's built-in `git-submodules` manager keeps the pointer current), plus a Jenkins shared-library reference (`@Library('ci-tools-lib')`).

## Core design rule

**All build logic lives in `*.just` modules. The Jenkinsfile and `vars/*.groovy` are glue only.**

Developers must reproduce full CI behaviour locally by running the same `just` recipes Jenkins runs. If logic leaks into Groovy, dev and CI diverge and "works on my machine" stops being meaningful.

- Acceptable in Groovy: `pipeline { ... }` declarative blocks, agent labels, `dir()`, `stash`/`unstash`, `withEnv`, `withCredentials`, `recordIssues`, `httpRequest`, `readJSON`/`writeJSON`, `currentBuild.description` flags, `env.*` reads.
- Not acceptable in Groovy: invoking scanners/linters/builders directly, computing versions, tagging/pushing images, rootfs extraction, file manipulation a developer would also need.
- A `sh` call in a Groovy wrapper should look like `sh "just <target> '${arg}'"` — nothing more.

## Architecture

Two layers:

1. **Just modules** (`*.just` at repo root, copied into consumers as `.ci/*.just`) — actual build logic. Composable via `import 'foo.just'`.
2. **Jenkins shared library** (`vars/*.groovy`) — thin per-stage wrappers + Jenkins-only concerns (changeset detection, recordIssues, PR build-file protection).

**Pipeline (declarative, in `vars/justContainer.groovy`):** Changeset → Prepare → Lint → Build → Test → Scan → Push → Cleanup.

- `currentBuild.description == 'SKIP'` is the cross-stage "no source changes, skip downstream" signal, set by `container.changeset` (the minimal first stage) when no changed files match `buildOnly` patterns (and neither the `forceBuild` config field nor the `FORCE_BUILD` build parameter is set). **A tagged release always builds:** `changeset` first calls `just container::_is-release` (HEAD exactly on a release tag matching `TAG_MATCH`) and returns early without setting SKIP when true, so releases are never skipped regardless of the changeset. A `.ci` submodule bump counts: Forgejo lists the gitlink as the bare path `.ci`, and `pathsChanged` tests `<path>/` too so a `\.ci/.*` pattern still matches. Gates Prepare, Lint, Build, Test, Scan, Push, cleanup — so the SKIP decision is made before any prep work runs.
- `Push` stage additionally gated by `not { changeRequest() }` — PRs never push.
- `justContainer` declares a `FORCE_BUILD` boolean parameter (default `false`), OR'd into `config.forceBuild` for the Changeset stage, so the skip gate can be overridden via the Jenkins "Build with Parameters" UI without editing config. (First build after introducing the parameter doesn't show the checkbox — registers from the second build onward.)

| File | Purpose |

Calls `notify.start(config)` unconditionally at the end of the Changeset stage and `notify.end(config)` in `post.always`; both self-filter inside `notify` (on `config.notify`, the events list and the `SKIP` flag), so notifications fire only when `config.notify` is set and `notifySkipped` governs start and end symmetrically. `post.cleanup` also calls `container.pruneSkippedBuilds(config)` to delete finished SKIP build records. |

Uniform application matters for env that feeds `git_tag`/`is_release` (`TAG_MATCH`) or image arch (`ARCH`): if it reached only Build, Push/Scan would compute a different tag; `lint` (`just scan-src` + recordIssues, then `just lint`); `build` (`just container::build` — runs only when prepare did not SKIP); `test` (`just test` if defined); `scan` (`just container::scan` + grype/betterleaks recordIssues); `push` (`just container::push <registry>` + `rm-remote-untagged`; `registry` required); `clean` (`just container::clean`); `cleanBuilder` (no-op unless `needBuilder`; calls `just clean-builder` — invoked from `post.cleanup`); `pruneSkippedBuilds` (walks `currentBuild.previousBuild` and deletes finished builds whose `description == 'SKIP'`, so monorepo history isn't cluttered by sibling jobs triggering on every push — **default on**, opt out with `discardSkipped: false`. A build can't delete *itself* — Jenkins refuses `Run.delete()` on a still-running build — so each build prunes the SKIP builds behind it and the latest SKIP lingers until the next run; safe because `disableConcurrentBuilds` means prior builds are done. Best-effort `try/catch` so a pending script approval for `RunWrapper.getRawBuild`/`Run.delete` logs a warning instead of failing the build; called last in `post.cleanup`). |
| `forgejo.groovy` | Forgejo REST client (formerly `gitea.groovy`; Forgejo is a Gitea fork sharing the `/api/v1` API + `token` auth, so the client is unchanged): parses `GIT_URL`, fetches PR files (`/pulls/N/files`) or commit diff (`/compare/base...head`), opens/reuses PRs (`openPullRequest`); `pathsChanged(files, patterns)` regex-matches — also tests each changed path with a trailing `/`, so a submodule/gitlink bump (Forgejo reports the bare path, e.g. `.ci`) matches a contents pattern like `\.ci/.*` |
| `protectBuild.groovy` | On PRs only (no-op otherwise): `git checkout origin/<target> -- <files>` to overwrite CI files (`.justfile`/`Jenkinsfile`/…) from the target branch, defending against a PR altering the CI that builds it. Wired via `container.prepare`'s `config.protectSubmodules` (`true` → `['.ci']`); opt-in because it also stops Renovate's submodule-bump PRs from validating the new pointer. Replaces the former `protectBuildFiles`. |
| `quietCheckout.groovy` | Manual `git fetch/checkout --quiet` replicating Git plugin behaviour with less console noise |
| `notify.groovy` | Optional build-lifecycle notifications (`start`/`end`) via an **apprise-api sidecar**. `end` maps `currentBuild.currentResult` → event (`success`/`failure`/`unstable`/`aborted`) + apprise `type`, self-filtering on the `notify.events` list and the `SKIP` flag. POSTs one JSON event (`title`/`body`/`type`/`tag`) via `httpRequest`; endpoint resolves as explicit `notify.key` → `${url}/notify/${key}` (key mode — destinations/secrets stored server-side in apprise-api), else `notify.urls` → `${url}/notify` (stateless), else `${url}/notify/default` (apprise-api's conventional `default` key). Validates args, writes the updates spec to a tmp JSON file, picks credential binding by repo URL scheme (SSH key via `sshagent`, HTTPS via `gitUsernamePassword` → `GIT_ASKPASS`), then `sh "just gitops::update ..."`. ## Conventions

### Shell escaping in Groovy `sh` calls

Always single-quote interpolated values: `sh "just <target> '${var}'"`. Use `withEnv` for environment variables instead of inline `KEY=VAL ...` so Jenkins sets them directly without shell parsing. Values come from the Jenkinsfile config map (controlled), so single-quote is safe; if a value could contain a `'`, escape it explicitly.

**Optional positional args must be omitted, not passed as `''`.** Just recipes typically default an arg like `image=git_repo_name`; passing an empty string from Groovy overrides that default with empty and breaks the recipe. Pattern:

```groovy
def imageArg = imageName ? " '${imageName}'" : ''
sh "just container::build${imageArg}"
```

### Optional just recipes

Stages call recipes conditionally so consumers don't need to define every target:

```
sh "if just --summary | grep -q lint; then just lint; fi"
```

### Builder container indirection

`needBuilder: true` in the config makes Prepare/Lint/Build run targets inside `local-{toolchain}-builder` via `just use-builder <target>`. The agent only needs Podman/Buildah, not language toolchains.

One working container is created per pipeline (named `local-<toolchain>-builder-<sanitized $BUILD_TAG>`) and reused across stages, so toolchain caches (cargo target, sccache) survive Prepare → Lint → Build → Test. `post.cleanup` calls `container.cleanBuilder(config)` which runs `just clean-builder` to remove it on any outcome. For the rust toolchain only, the sccache cache dir on the agent (`$SCCACHE_DIR` or `~/.cache/sccache`) is bind-mounted into the container, so the cache survives `clean-builder` and is shared with anything else on the agent using sccache.

### Multi-arch

Per-arch images tagged `<image>:<tag>-<arch>`, then `push` builds a manifest list referencing both archs. `all_archs := "amd64 arm64"` in `git.just`.

### Versioning

`git_tag` from `git describe --tags --match "<tag_match>" --dirty`, falls back to short SHA, where `tag_match := env_var_or_default("TAG_MATCH", "v*.*.*")`. The default match `v*.*.*` covers single-repo projects; monorepo services override it so `git describe` only considers that service's release tags — but **via the `TAG_MATCH` env var, not an in-justfile `export`**: Jenkins consumers set `config.env: ['TAG_MATCH=<service>/v*.*.*']` (applied through `withEnv` around every stage's just shells, so the tag is consistent across Changeset/Build/Scan/Push), local dev uses `TAG_MATCH=<service>/v*.*.* just …` or `just tag_match='<service>/v*.*.*' …`. An in-justfile `export TAG_MATCH := …` does **not** work: just propagates neither `{{ }}` nor `export`s into the assignment-time `shell()` subshells where `git_tag`/`is_release` are computed, and an imported `tag_match` can't be redefined. On non-main branches, `tag` becomes `<git_tag>-<sanitized-branch>` unless the branch is already a substring/equal.

`git_tag`/`git_branch`/`git_repo_name` each anchor git to the **consumer (super)project**, not `.ci`: just cd's into a module's directory and `.ci` is a submodule with its own `.git`, so a bare `git describe`/`config` run there would describe `ci-tools-lib` instead of the consumer (symptom: image `…/ci-tools-lib`, tag a short SHA). A single `root` variable resolves it as `git rev-parse --show-superproject-working-tree` (non-empty only inside a submodule) → else `--show-toplevel`; each git fact (`git_tag`/`git_branch`/`git_repo_name`/`is_release`) then runs `git -C "$1" …` with `root` (and, for the tag facts, `tag_match`) threaded in as positional args `$1`/`$2` via just's `shell()` function — correct whether evaluated from `.ci`, the consumer root, or the library's own build. `shell()` (just >= 1.15) is the only way to share these: just interpolates neither `{{ }}` nor `export`ed variables into assignment-time backtick/`shell()` subshells (verified — an in-justfile `export TAG_MATCH := …` reaches recipe bodies but **not** these assignments, which is why `TAG_MATCH` overriding goes through the env var / CLI, see Versioning). Same rule governs the `IMAGE_NAME` override across the `mod container` boundary: a plain/`export`ed var in the consumer's `.justfile` is inert, but `set dotenv-load` + a `.env` (`IMAGE_NAME=<svc>`) **does** reach the submodule's assignment-time `env_var_or_default` (verified — dotenv is loaded into the environment before variable evaluation), as do a real launch env var and the positional arg.

### Monorepo support

The library is path-aware so a single `.ci/` submodule at the repo root serves multiple services in subdirectories:

- `builder.just` resolves `Dockerfile.<toolchain>` via `source_directory()` so `update-builder` works regardless of caller cwd.
- `use-builder` mounts the repo root (not `$(pwd)`) and `cd`s to the caller's relative path inside the container, so `import '../../.ci/<lang>.just'` from a service justfile resolves at runtime.
- `git.just` honours the `TAG_MATCH` env var (or `just tag_match=…`) for per-service tag prefixes — not an in-justfile `export` (see Versioning).
- `container.just` derives the image name from `IMAGE_NAME` (`image_name := env_var_or_default("IMAGE_NAME", git_repo_name)`) so per-service builds don't inherit the monorepo repo name. Same `mod`-boundary rule as `TAG_MATCH`: set it via `set dotenv-load` + a `.env` (`IMAGE_NAME=<svc>`), a real launch env var, or `config.env: ['IMAGE_NAME=<svc>']` — **not** a plain/`export`ed var in the consumer `.justfile` (inert across `mod`). Jenkins already passes it positionally via `imageName`. `container.just`'s `registry` positional follows the same pattern via `REGISTRY` (`default_registry := env_var_or_default("REGISTRY", "")`) — set it in the same `.env`; Jenkins still forwards the `registry:` config field positionally.
- `container.prepare` defaults `protect` to `["${workDir}/.justfile", "${workDir}/Jenkinsfile"]` — service-scoped without needing per-Jenkinsfile config. `.ci` is not in the default: a plain file checkout can't restore a submodule (the parent repo tracks only its gitlink + `.gitmodules`). Restoring the `.ci` pointer on PR builds is opt-in via `config.protectSubmodules` (`true` → `['.ci']`); `protectBuild` then restores the gitlink + `.gitmodules` from target and re-syncs the working tree. Left off (default), Renovate's `.ci`-bump PRs stay able to validate the new pointer; turn it on for repos that accept untrusted fork PRs. Note: `Jenkinsfile` protection only matters for *trusted* PRs — for untrusted fork PRs the branch source's trust strategy already takes the Jenkinsfile from the target branch, while for trusted/same-repo PRs Jenkins reads the PR's Jenkinsfile before any step runs, so the real boundary for external contributors is Jenkins controller PR-trust policy. Either way the PR's *workspace* files (`.justfile`, `.ci`) are always the PR's version regardless of trust — that gap is exactly what `protectBuild` closes.

Per-service Jenkinsfile typically sets `workDir`, `imageName`, `buildOnly` (regex with the service's path prefix), and `needBuilder`. See README "Monorepo layout" for a full example.

## External integration

- **Forgejo** — changeset detection via REST API. Forgejo is a hard fork of Gitea and keeps the same `/api/v1` REST API and `token <token>` auth, so the `vars/forgejo.groovy` client (formerly `gitea.groovy`) works unchanged. Credentials: Jenkins username/password credential ID `gitea-jenkins-password` (default — a consumer-provisioned Jenkins credential whose ID is kept for back-compat; override via `config.credentialsId`). API base derived from `env.GIT_URL`, else `env.FORGEJO_URL` (falling back to `env.GITEA_URL`), or overridden via config.
- **GitOps writeback** — `updateGitops(...)` from a consumer Jenkinsfile commits image tag/digest changes to a Forgejo-hosted manifests repo so ArgoCD/Flux pick them up. Two modes: `push` (direct commit to base branch) or `pr` (force-with-lease to a PR branch + idempotent Forgejo PR open). Credentials: SSH-key credential when `repo` is `git@host:...` / `ssh://...`, username/password credential when `repo` is `https://...` (uses `gitUsernamePassword` so the token never lands in the URL or `.git/config`). PR mode additionally needs a username/password credential as the Forgejo API token. Consumers must `mod gitops '.ci/gitops.just'` so the recipes are reachable. Examples: `examples/Jenkinsfile.gitops-{push,pr}.groovy`.
- **AWS ECR (public or private)** — `aws ecr-public get-login-password` (region `us-east-1`, fixed for ECR Public) or `aws ecr get-login-password` (region parsed from the registry hostname `*.dkr.ecr.<region>.amazonaws.com`) piped to `podman login`. The `registry` positional defaults to the `REGISTRY` env var (`default_registry := env_var_or_default("REGISTRY", "")`), else must be declared per consumer. Jenkins consumers pass `registry: '...'` in `justContainer(...)`; `container.push` validates and forwards it as the first positional arg to the just recipes. Local dev consumers pass it explicitly, set `REGISTRY` (service `.env` / launch env), or wrap the recipes in their own justfile. Local dev and Jenkins agent both rely on ambient AWS credentials (env vars, instance profile, etc.) — no credential plumbing in the library.
- **Build notifications (apprise-api)** — optional. `notify.groovy` POSTs build-lifecycle events to an [apprise-api](https://github.com/caronc/apprise-api) sidecar running next to the Jenkins controller, which fans out to chat targets (Slack/Matrix/Mattermost/Teams). Endpoint from `env.APPRISE_API_URL` (controller-global) or `notify.url`. Chat destinations + their secrets live in apprise-api **config keys** (referenced via `notify.key`, defaulting to the `default` key when omitted), not in this repo. Opt-in per consumer via the `notify` config map; uses the existing HTTP Request plugin (no new plugin). See README "Build notifications".
- **Required Jenkins plugins:** Pipeline (declarative), Git, HTTP Request, Pipeline Utility Steps, Credentials Plugin, Warnings Next Generation (`recordIssues` with `grype` and `sarif` tools).

## Known gaps

- **No tests for the library itself.** No Jenkinsfile linter integration. Correctness verified by running against real consumer projects.

## Working with this repo

- Edits to `vars/*.groovy` only affect Jenkins consumers when the library is reloaded (controller-side cache).
- **Trusted vs sandboxed library.** Groovy-sandbox constraints in `vars/*.groovy` (no `<<`/`new`, script-approval prompts for `RunWrapper.getRawBuild`/`Run.delete`, the red "dangerous signature" flag) apply only when the library is loaded **untrusted** (folder-scoped, or the untrusted-global section). Loading ci-tools-lib as a **trusted** Global Pipeline Library (Manage Jenkins → System) runs it unsandboxed — no approvals, `pruneSkippedBuilds` works without the red `getRawBuild` — and is orthogonal to the `.ci` submodule (the controller fetches the library from its own SCM config, not the workspace submodule). Trust does **not** relax CPS constraints (`.sort{closure}` still breaks) or Job-DSL expressiveness (`GiteaSCMSource` has no `@Symbol` → still needs `configure{}` raw XML in `seedMultiJob`).
- Edits to `*.just` reach consumers when their `.ci/` submodule pointer is bumped — automatically via Renovate's `git-submodules` manager, or manually via `git submodule update --remote .ci` / the `ci-pull-upstream` recipe.
- The library location is consumer-configurable for mirroring: the submodule URL (any mirror) lives in the consumer's `.gitmodules` — nothing hardcodes the canonical URL. `examples/migrate-to-submodule.sh` honours `CI_TOOLS_URL`/`CI_TOOLS_BRANCH`.
- Don't add a recipe to a `*.just` module without considering whether developers will actually run it locally; if it's Jenkins-only, it belongs in Groovy.
- Don't add `sh` logic to Groovy beyond invoking `just`. If the temptation arises, the right move is a new just recipe.
